================================================================================
DADOS E ETL — responsabilidade do PostgreSQL (Supabase / DW)
================================================================================

O que fica AQUI (versionamento no Git):
  • Scripts SQL de referência: views, materialized views, funções, grants.
  • Coloque arquivos em postgres/sql/ (ex.: 001_vw_kpi.sql).
  • A execução real do ETL/carga é no ambiente do banco (Airbyte, dbt, job
    agendado, etc.) — não no Node.js.

Regra de ouro:
  • Cálculo pesado, agregação, join grande, janela temporal, deduplicação:
    tudo no Postgres (ou ferramenta de ETL que alimenta o Postgres).
  • A view exposta ao app já deve devolver linhas no formato “pronto para o
    painel” (colunas com nomes estáveis), para o BI não reprocessar no API.

================================================================================
Node.js (Express) — fino e rápido
================================================================================

  • Só: SELECT em views via Supabase (db.js → fetchView).
  • Filtros de tela (regional, unidade, período) viram .eq / .gte na query.
  • Mapeamento permitido: renomear campos, tipos simples, montar objeto JSON
    a partir de cada LINHA — complexidade O(n) nas linhas retornadas.
  • Evitar: re-agregar no JavaScript, simular GROUP BY, carregar fatos brutos
    gigantes para processar no servidor — isso gera delay e carga desnecessária.

================================================================================
React
================================================================================

  • Consome { ok, data }.
  • A escolha de gráfico (pipeline/viz.mjs) é heurística leve no cliente —
    não substitui ETL nem substitui views bem modeladas no Postgres.

================================================================================
API Gerência (contratos JSON — dados zerados até as views)
================================================================================

  GET /api/v1/gerencia/unidades-ps
    → data[] : { unidadeId, unidadeNome, regional }  (só unidades com PS)

  GET /api/v1/gerencia/metas-por-volumes
    → matriz “Metas por volumes”: months[], mesKeys[], data[] (indicadores +
      m1/m2/m3/t + subItems[] por unidade no drill)

  GET /api/v1/gerencia/metas-por-volumes/indicador/:indicadorKey/unidades
    → drill opcional por indicador: { indicadorKey, indicadorNome, unidades[] }

  GET /api/v1/overview/metas-volumes  (alias do mesmo payload da matriz)

  GET /api/v1/gerencia/metricas-por-unidade
    → colunas[] : { key, label, kind, pctSense?, pctGreenAt?, pctRedAt? }
       kind pct: pctSense = high_good | low_good (verde/vermelho na UI).
       Limiares podem vir da view por período quando o ETL os definir.

  GET /api/v1/gerencia/metas-acompanhamento-gestao?metric=&period=&regional=&unidade=
    → Painel “Metas de acompanhamento”: catalog[], gauge (global), metaRibbon,
      months[], series[] por unidade (cor fixa). Valores zerados até a view
      Postgres preencher.

  GET /api/v1/gerencia/metas-conformes-por-unidade?period=&regional=&unidade=
    → Gráfico “% metas conformes por unidade”: months[], isPercent, series[].
      Zerado até a view Postgres; só filtros globais da tela.
